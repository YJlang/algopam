# [알고팜 2주차] BOJ 1822 차집합 - list에서 시간초과가 난 이유를 set으로 이해하기

이번 문제는 처음에는 "차집합이니까 A를 돌면서 B에 없는 값만 고르면 되겠지" 하고 접근했다. 아이디어 자체는 맞았는데, `list`에서 `in`을 쓴 부분 때문에 시간초과가 났다. 이번 글에서는 내가 처음 짰던 풀이가 왜 느렸는지, 그리고 왜 `set`으로 바꾸면 훨씬 빨라지는지를 같이 정리해보려고 한다.

문제 링크: https://www.acmicpc.net/problem/1822

## 문제 정보

- 시간 제한: 2초
- 메모리 제한: 256MB
- 풀이 감각: 자료구조, 집합, 시간복잡도 감각 익히기

## 문제 설명

문제 원문은 위 링크에서 확인할 수 있고, 아래는 풀이를 위해 다시 정리한 내용이다.

- 자연수로 이루어진 두 집합 `A`, `B`가 주어진다.
- 이때 `A`에는 있지만 `B`에는 없는 원소들만 골라야 한다.
- 결과는 원소의 개수와 함께, 증가하는 순서로 출력해야 한다.
- 만약 그런 원소가 하나도 없으면 `0`만 출력하면 된다.

## 입력 형식

- 첫째 줄에 집합 `A`의 크기 `n`, 집합 `B`의 크기 `m`이 주어진다.
- 둘째 줄에는 집합 `A`의 원소들이, 셋째 줄에는 집합 `B`의 원소들이 공백으로 주어진다.
- `1 ≤ n, m ≤ 500,000`이고, 각 원소는 `2,147,483,647` 이하의 자연수이며 같은 집합 안에서는 중복되지 않는다.

## 출력 형식

- 첫째 줄에 `A - B`의 원소 개수를 출력한다.
- 둘째 줄에는 그 원소들을 공백으로 구분해 증가하는 순서로 출력한다. 원소가 없으면 둘째 줄은 출력하지 않아도 된다.

## 예제 입력 1

```text
4 3
2 5 11 7
9 7 4
```

## 예제 출력 1

```text
3
2 5 11
```

## 예제 입력 2

```text
3 5
2 5 4
1 2 3 4 5
```

## 예제 출력 2

```text
0
```

## 접근 아이디어

1. `A`에 있는 수를 하나씩 보면서, 그 수가 `B`에 없으면 결과에 넣으면 된다.
2. 그런데 "없다"를 검사하는 자료형이 `list`인지 `set`인지에 따라 속도 차이가 아주 크게 난다.
3. 출력은 오름차순이어야 하므로, `A`를 정렬한 뒤 걸러내거나 마지막에 결과를 정렬하면 된다.

## 내가 처음 작성한 코드

```python
import sys

input = sys.stdin.readline

n, m = map(int, input().split())
a = list(map(int, input().split()))
b = list(map(int, input().split()))

a.sort()
b.sort()

result = []

for i in range(n):
    if a[i] not in b:
        result.append(a[i])

print(len(result))
for x in result:
    print(x, end=" ")
```

## 이 코드에서 내가 하려고 했던 것

방향 자체는 자연스러웠다. `A`와 `B`를 읽고, `A`의 각 원소가 `B` 안에 없으면 결과 리스트에 담는 방식이다. 출력이 증가하는 순서여야 해서 `a.sort()`를 한 것도 맞는 선택이었다.

즉, "정렬된 `A`를 앞에서부터 보면서 `B`에 없는 값만 남기자"라는 흐름으로 풀고 싶었던 코드였다.

## 아쉬웠던 점

문제는 이 줄이었다.

```python
if a[i] not in b:
```

여기서 `b`가 `list`이기 때문에, 파이썬은 `a[i]`가 있는지 확인하려고 `b`를 앞에서부터 하나씩 비교한다. 찾으면 멈추지만, 없거나 뒤쪽에 있으면 거의 끝까지 다 봐야 한다.

즉, `a`를 한 번 도는 동안 `n`번 검사하고, 검사 한 번이 최악 `m`칸까지 갈 수 있어서 전체가 사실상 `O(nm)`이 된다. `n`과 `m`이 둘 다 최대 500,000까지 갈 수 있으니 이 방식은 금방 시간초과가 난다.

또 하나 헷갈렸던 점은 `b.sort()`를 해도 `x in b`가 빨라지지는 않는다는 것이다. `list`에서 `in`은 정렬 여부와 상관없이 보통 처음부터 끝까지 순서대로 검사한다. 정렬의 이점을 쓰려면 이진 탐색을 직접 써야 한다.

```text
예를 들어 n = 500,000, m = 500,000이고
A와 B가 거의 겹치지 않으면,
A의 각 원소마다 B를 길게 훑게 되어 매우 느려진다.
```

## 개선 풀이 1

```python
import sys

input = sys.stdin.readline

n, m = map(int, input().split())
a = list(map(int, input().split()))
b = set(map(int, input().split()))

a.sort()
result = [x for x in a if x not in b]

print(len(result))
if result:
    print(*result)
```

내가 최종적으로 고친 방향은 `B`를 `set`으로 바꾸는 것이었다. `set`은 포함 여부 검사인 `x in b`가 평균적으로 `O(1)`이라서, `A`를 한 번 돌면서 바로바로 걸러낼 수 있다.

이 풀이에서 중요한 포인트는 두 가지다. 첫째, 출력을 오름차순으로 만들기 위해 `a.sort()`를 먼저 한다. 둘째, `x not in b`에서의 `b`는 `list`가 아니라 `set`이므로 한 번 검사할 때 훨씬 빠르다.

참고로 이 풀이를 적을 때 헷갈리기 쉬운 부분이 있는데, `result = [x for x in a if x not in b]`의 복잡도는 `O(m)`이 아니라 `O(n)`에 가깝다. 이유는 `a`를 한 번 순회하고, 각 원소에 대해 `set` 조회를 평균 `O(1)`로 하기 때문이다.

## 개선 풀이 2

```python
import sys

input = sys.stdin.readline

n, m = map(int, input().split())
a = set(map(int, input().split()))
b = set(map(int, input().split()))

result = sorted(a - b)

print(len(result))
if result:
    print(*result)
```

이 방법은 파이썬의 집합 연산을 그대로 쓰는 풀이이다. `a - b`가 바로 차집합이므로 코드가 아주 짧고 의도가 잘 보인다.

다만 출력이 증가하는 순서여야 하므로 마지막에 `sorted()`를 한 번 써야 한다. 개인적으로는 "차집합"이라는 문제 이름과 가장 잘 맞는 표현이라서, 개념 복습용으로는 이 풀이도 꽤 마음에 들었다.

## 시간 복잡도

- 내가 처음 작성한 풀이: `O(n log n + m log m + nm)`, 지배적인 항은 `O(nm)`
- 개선 풀이 1: `O(n log n + m)`
- 개선 풀이 2: `O(n + m + k log k)` (`k`는 `A - B`의 원소 개수, 최악에는 `n`)

처음 풀이에서는 `a.sort()`와 `b.sort()`가 각각 `O(n log n)`, `O(m log m)`이고, 가장 큰 문제는 `a`의 각 원소마다 `b`를 리스트 탐색하는 부분이다. 리스트에서 `in`은 최악 `O(m)`이므로 이 검사를 `n`번 하면 `O(nm)`이 된다.

개선 풀이 1에서는 `b`를 `set`으로 만드는 데 `O(m)`, `a.sort()`에 `O(n log n)`, 그리고 `a`를 한 번 돌며 `set` 조회를 하는 데 `O(n)`이 든다. 그래서 전체는 `O(n log n + m)`으로 정리할 수 있다.

개선 풀이 2에서는 `a`, `b`를 집합으로 만드는 데 `O(n + m)`, 차집합 계산에 `O(n)`, 결과 정렬에 `O(k log k)`가 걸린다. 결과가 많아도 최악 `k ≤ n`이므로 충분히 빠르다.

## 이번 문제에서 배운 점

- `list`를 정렬했다고 해서 `x in list`가 빨라지는 것은 아니다.
- "포함 여부를 많이 검사해야 하는 문제"에서는 `set`을 먼저 떠올리는 습관이 정말 중요하다.
- 시간복잡도는 한 줄만 보지 말고, 그 연산이 몇 번 반복되는지까지 같이 봐야 정확하게 잡힌다.
