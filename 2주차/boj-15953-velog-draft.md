# [알고팜 2주차] BOJ 15953 상금 헌터 - 조건문으로 맞췄지만 표로 보면 더 깔끔하다

이번 문제는 등수 구간마다 상금이 정해져 있어서, 표를 코드로 어떻게 옮길지가 핵심이었다. 처음 작성한 풀이는 `if/elif`로 순위 구간을 직접 나눴고, 문제 조건에서는 잘 동작한다. 다만 같은 패턴이 1회 대회와 2회 대회에 반복되기 때문에, 표 형태로 정리하면 더 읽기 좋고 수정하기도 쉬워진다.

문제 링크: https://www.acmicpc.net/problem/15953

## 문제 정보

- 시간 제한: 1초 (추가 시간 없음)
- 메모리 제한: 512MB
- 풀이 감각: 구현, 조건 분기, 표를 코드로 옮기기

## 문제 설명

문제 원문은 위 링크에서 확인할 수 있고, 아래는 풀이를 위해 다시 정리한 내용이다.

- 제이지가 두 번의 카카오 코드 페스티벌에서 받을 수 있는 총 상금을 계산해야 한다.
- 2017년 대회는 1등부터 6등 구간까지, 총 21명에게 상금을 준다.
- 2018년 대회는 1등부터 5등 구간까지, 총 31명에게 상금을 준다.
- 각 가정마다 2017년 순위 `a`, 2018년 순위 `b`가 주어지고, 두 대회의 상금을 합산해 출력하면 된다.

2017년 상금 구간은 누적 순위로 보면 이렇게 정리할 수 있다.

```text
1등          -> 5,000,000원
2등 ~ 3등    -> 3,000,000원
4등 ~ 6등    -> 2,000,000원
7등 ~ 10등   ->   500,000원
11등 ~ 15등  ->   300,000원
16등 ~ 21등  ->   100,000원
```

2018년 상금 구간은 이렇게 된다.

```text
1등          -> 5,120,000원
2등 ~ 3등    -> 2,560,000원
4등 ~ 7등    -> 1,280,000원
8등 ~ 15등   ->   640,000원
16등 ~ 31등  ->   320,000원
```

## 입력 형식

- 첫 번째 줄에 가정의 개수 `T`가 주어진다.
- 다음 `T`개 줄에는 각 가정의 순위 `a`, `b`가 공백으로 구분되어 주어진다.
- `a`는 `0` 이상 `100` 이하, `b`는 `0` 이상 `64` 이하이다.

## 출력 형식

- 각 가정마다 받을 총 상금을 한 줄에 하나씩 출력한다.
- 출력은 입력으로 들어온 가정의 순서를 유지해야 한다.

## 예제 입력 1

```text
6
8 4
13 19
8 10
18 18
8 25
13 16
```

## 예제 출력 1

```text
1780000
620000
1140000
420000
820000
620000
```

## 접근 아이디어

1. 각 대회별로 순위가 어떤 상금 구간에 들어가는지 확인한다.
2. 순위가 `0`이거나 상금권 밖이면 상금은 `0`원이다.
3. 2017년 상금과 2018년 상금을 더해서 출력한다.

예를 들어 예제의 첫 줄 `8 4`를 보면, 2017년 순위 `8`은 `7등 ~ 10등` 구간이므로 `500,000원`이다. 2018년 순위 `4`는 `4등 ~ 7등` 구간이므로 `1,280,000원`이다. 둘을 더하면 `1,780,000원`이 된다.

## 내가 처음 작성한 코드

아래 코드는 블로그에서 읽기 쉽도록 필요한 설명 주석을 덧붙여 정리한 버전이다.

```python
# 2주차 과제 BOJ 15953: 상금 헌터
# https://www.acmicpc.net/problem/15953
import sys

input = sys.stdin.readline

t = int(input())

# 각 구간의 상금을 순서대로 저장한다.
a_prize_list = [5000000, 3000000, 2000000, 500000, 300000, 100000]
b_prize_list = [5120000, 2560000, 1280000, 640000, 320000]

for _ in range(t):
    a, b = map(int, input().split())
    a_prize = 0
    b_prize = 0

    # 2017년 대회 순위 구간 확인
    if a == 1:
        a_prize = a_prize_list[0]
    elif 1 < a <= 3:
        a_prize = a_prize_list[1]
    elif 3 < a <= 6:
        a_prize = a_prize_list[2]
    elif 6 < a <= 10:
        a_prize = a_prize_list[3]
    elif 10 < a <= 15:
        a_prize = a_prize_list[4]
    elif 15 < a <= 21:
        a_prize = a_prize_list[5]

    # 2018년 대회 순위 구간 확인
    if b == 1:
        b_prize = b_prize_list[0]
    elif 1 < b <= 3:
        b_prize = b_prize_list[1]
    elif 3 < b <= 7:
        b_prize = b_prize_list[2]
    elif 7 < b <= 15:
        b_prize = b_prize_list[3]
    elif 15 < b <= 31:
        b_prize = b_prize_list[4]

    print(a_prize + b_prize)
```

이 코드는 예제 입력을 넣었을 때 공식 예제 출력과 똑같이 나온다. 순위가 상금권 밖이면 `a_prize`, `b_prize`가 처음 값인 `0`으로 남기 때문에 자연스럽게 상금이 없는 경우도 처리된다.

## 이 코드에서 내가 하려고 했던 것

처음 풀이는 문제의 상금표를 그대로 조건문으로 옮긴 방식이다.

`a_prize_list`에는 2017년 상금이 높은 순서대로 들어 있고, `b_prize_list`에는 2018년 상금이 높은 순서대로 들어 있다. 그다음 `a`와 `b`가 어느 구간에 들어가는지 `if/elif`로 확인해서 해당 상금을 꺼낸다.

예를 들어 `elif 6 < a <= 10`은 `7등부터 10등까지`라는 뜻이다. 이 구간의 상금은 `a_prize_list[3]`, 즉 `500000원`이다.

## 아쉬웠던 점

정답은 맞지만, 구조적으로는 반복이 많다.

상금표 자체에는 "몇 등까지 이 상금인가"라는 정보와 "상금이 얼마인가"라는 정보가 같이 있다. 그런데 원래 코드에서는 상금은 리스트에 있고, 순위 구간은 조건문에 따로 흩어져 있다. 그래서 나중에 표를 수정해야 한다면 리스트와 조건문을 함께 확인해야 한다.

또 2017년 대회와 2018년 대회의 처리 방식이 거의 같은데, 코드가 각각 따로 적혀 있다. 이런 경우에는 "순위 하나를 넣으면 상금을 돌려주는 함수"로 빼면 흐름이 조금 더 단순해진다.

```text
틀리는 반례라기보다는 점검하면 좋은 입력:
1
0 0

출력은 0이어야 한다.
원래 코드에서는 두 상금 변수가 0으로 시작하고 어떤 조건에도 걸리지 않으므로 올바르게 0을 출력한다.
```

## 개선 풀이 1

아래 코드는 변수와 핵심 동작이 눈에 들어오도록 짧은 주석을 덧붙인 버전이다.

```python
import sys

input = sys.stdin.readline

# (상금을 받는 마지막 순위, 상금)
FIRST_FESTIVAL = [
    (1, 5000000),
    (3, 3000000),
    (6, 2000000),
    (10, 500000),
    (15, 300000),
    (21, 100000),
]

SECOND_FESTIVAL = [
    (1, 5120000),
    (3, 2560000),
    (7, 1280000),
    (15, 640000),
    (31, 320000),
]


def get_prize(rank, table):
    # rank가 0이면 본선에 진출하지 못한 경우라 상금이 없다.
    if rank == 0:
        return 0

    # 누적 순위의 마지막 값 안에 들어오면 해당 상금을 받는다.
    for last_rank, prize in table:
        if rank <= last_rank:
            return prize

    # 상금권 밖 순위도 상금이 없다.
    return 0


t = int(input())

for _ in range(t):
    a, b = map(int, input().split())
    print(get_prize(a, FIRST_FESTIVAL) + get_prize(b, SECOND_FESTIVAL))
```

이 버전은 상금표를 `(마지막 순위, 상금)` 형태로 묶었다. 예를 들어 `(10, 500000)`은 "10등까지는 500,000원을 받을 수 있는 구간"이라는 뜻이다.

`get_prize(rank, table)`은 순위를 하나 받고, 표를 앞에서부터 보면서 처음으로 `rank <= last_rank`가 되는 상금을 반환한다. 만약 끝까지 못 찾으면 상금권 밖이므로 `0`을 반환한다.

## 개선 풀이 2

아래 코드는 블로그 독자가 흐름을 따라가기 쉽도록 설명 주석을 덧붙인 버전이다.

```python
import sys

input = sys.stdin.readline

# rank를 바로 인덱스로 쓰기 위해 0등 자리도 0원으로 둔다.
first_prizes = (
    [0]
    + [5000000] * 1
    + [3000000] * 2
    + [2000000] * 3
    + [500000] * 4
    + [300000] * 5
    + [100000] * 6
    + [0] * (100 - 21)
)

second_prizes = (
    [0]
    + [5120000] * 1
    + [2560000] * 2
    + [1280000] * 4
    + [640000] * 8
    + [320000] * 16
    + [0] * (64 - 31)
)

t = int(input())

for _ in range(t):
    a, b = map(int, input().split())
    print(first_prizes[a] + second_prizes[b])
```

이 버전은 각 순위의 상금을 미리 리스트로 펼쳐놓는다. 그러면 `first_prizes[8]`처럼 순위를 그대로 인덱스로 사용해서 상금을 바로 꺼낼 수 있다.

예를 들어 `first_prizes[8]`은 2017년 8등의 상금인 `500000`이고, `second_prizes[4]`는 2018년 4등의 상금인 `1280000`이다.

주의할 점은 `a`와 `b`가 `0`일 수 있다는 것이다. 그래서 리스트의 0번 인덱스도 `0`원으로 만들어 두면, 진출하지 못한 경우도 자연스럽게 처리된다.

## 시간 복잡도

- 내가 처음 작성한 풀이: `O(T)`
- 개선 풀이 1: `O(T)`
- 개선 풀이 2: `O(T)`

여기서 `T`는 입력으로 들어오는 가정의 개수다. 이 문제에서는 `1 ≤ T ≤ 1,000`이다.

최악의 경우는 `T`가 1,000이고, 매 줄마다 상금권 밖 순위가 들어와서 조건을 끝까지 확인하는 상황으로 볼 수 있다. 예를 들어 `a = 100`, `b = 64`라면 원래 코드는 2017년 조건 6개와 2018년 조건 5개를 모두 확인한 뒤 상금이 없다고 판단한다.

그래도 한 가정마다 확인하는 조건의 개수는 최대 11개 정도로 고정되어 있다. 입력이 10줄이면 이 고정 작업을 10번 하고, 입력이 1,000줄이면 1,000번 하는 구조다. 그래서 전체 시간은 `T`에 비례하고, Big-O로는 `O(T)`다.

개선 풀이 1도 `get_prize()` 안에서 보는 표의 길이가 2017년은 6칸, 2018년은 5칸으로 고정되어 있다. 따라서 한 가정당 `O(1)`, 전체는 `O(T)`다.

개선 풀이 2는 미리 만들어 둔 리스트에서 `first_prizes[a]`, `second_prizes[b]`를 바로 꺼낸다. 인덱스로 리스트 값을 꺼내는 작업은 한 번에 끝나는 `O(1)`이고, 이 작업을 `T`번 반복하므로 전체는 `O(T)`다.

조금 더 쉽게 말하면, 이 문제는 순위 범위가 작고 고정되어 있어서 한 사람의 상금을 계산하는 데 드는 일은 거의 일정하다. 입력 줄이 2배가 되면 계산해야 하는 가정 수도 2배가 되므로 실행 시간도 대략 2배가 된다.

세 풀이 모두 Big-O는 `O(T)`로 같지만, 읽기 쉬운 방향은 조금 다르다. 원래 풀이는 조건이 눈에 바로 보이는 장점이 있고, 개선 풀이 1은 상금표와 계산 로직이 분리되어 수정하기 쉽다. 개선 풀이 2는 순위를 인덱스로 바로 조회하므로 코드가 짧고 빠르지만, 리스트를 펼치는 방식이 처음 보면 약간 낯설 수 있다.

## 이번 문제에서 배운 점

- 문제의 표를 코드로 옮길 때는 조건문으로 직접 나눌 수도 있고, 표 자체를 데이터로 만들어 처리할 수도 있다.
- 상금권 밖 순위와 `0`처럼 예외처럼 보이는 값도 초기값이나 리스트의 0번 인덱스로 자연스럽게 처리할 수 있다.
- 반복문 안의 조건이 여러 개여도 그 개수가 고정되어 있으면 한 입력당 `O(1)`, 전체는 입력 줄 수에 비례해 `O(T)`가 된다.
