# [알고팜 2주차] BOJ 10250 ACM 호텔 - `W`보다 중요한 건 층과 순서였다

처음 보면 호텔 크기가 `H x W`라서 `W`도 뭔가 적극적으로 써야 할 것처럼 보인다.  
그런데 실제로는 손님을 배정하는 순서만 이해하면, 핵심 계산에는 `H`와 `N`이 더 중요하다.  
이번 글에서는 내가 쓴 정답 코드를 기준으로, 왜 이 공식이 맞는지와 `n % h == 0` 예외를 왜 따로 처리해야 하는지 정리해봤다.

문제 링크: https://www.acmicpc.net/problem/10250

## 문제 설명

호텔은 `H`층 건물이고 각 층마다 `W`개의 방이 있다.  
손님은 엘리베이터에서 걷는 거리가 가장 짧은 방을 선호하므로, 같은 호수라면 아래층부터 차례대로 배정된다.

- 1번째 손님은 101호
- 2번째 손님은 201호
- 3번째 손님은 301호
- ...
- `H`번째 손님까지는 1호 라인을 아래층부터 위층까지 채운다
- 그다음부터는 2호 라인으로 넘어간다

즉, 방을 가로로 채우는 게 아니라 세로로 채운다고 생각하면 된다.

## 입력 형식

- 첫째 줄에 테스트 케이스 개수 `T`가 주어진다
- 각 테스트 케이스마다 `H`, `W`, `N`이 공백으로 구분되어 주어진다
- `H`는 층 수, `W`는 각 층의 방 수, `N`은 `N`번째 손님을 뜻한다

## 출력 형식

- 각 테스트 케이스마다 `N`번째 손님이 배정받을 방 번호를 출력한다

## 예제 입력 1

```text
2
6 12 10
30 50 72
```

## 예제 출력 1

```text
402
1203
```

## 접근 아이디어

1. 손님은 같은 호수에서 아래층부터 위층까지 먼저 채운다.
2. 그래서 `N`번째 손님의 층수는 `N % H`와 관련 있다.
3. 호수는 `N // H`와 관련 있고, `N`이 `H`의 배수일 때만 예외 처리가 필요하다.

예를 들어 `H = 6`일 때 손님 배정 순서를 써보면 이렇게 된다.

```text
1번째 -> 101
2번째 -> 201
3번째 -> 301
4번째 -> 401
5번째 -> 501
6번째 -> 601
7번째 -> 102
8번째 -> 202
9번째 -> 302
10번째 -> 402
```

이 패턴만 보이면,

- 층수는 `H`개를 한 묶음으로 돌고
- 호수는 `H`명씩 찰 때마다 1씩 증가

한다는 걸 알 수 있다.

## 내가 처음 작성한 코드

아래 코드는 제출한 로직을 블로그용으로 주석만 조금 다듬은 버전이다.

```python
import sys

input = sys.stdin.readline

t = int(input())  # 테스트 케이스 개수

for _ in range(t):
    h, w, n = map(int, input().split())  # h: 층 수, w: 각 층의 방 수, n: n번째 손님

    if n % h == 0:
        print(h * 100 + n // h)
    else:
        print(n % h * 100 + n // h + 1)
```

## 이 코드에서 내가 하려던 것

이 코드는 손님을 실제로 하나씩 시뮬레이션하지 않고, 규칙을 바로 식으로 바꿔서 계산한다.

- `n % h`로 몇 층에 배정되는지 계산한다
- `n // h`로 몇 호 라인까지 찼는지 계산한다
- 다만 `n`이 `h`의 배수라면 층수가 `0`이 되어버리므로 별도 처리한다

즉, 이 풀이는 "패턴을 발견해서 바로 계산하는 풀이"다.  
이 문제에서는 이런 방식이 가장 깔끔하다.

## 아쉬웠던 점 또는 주의할 점

내 코드는 정답이고, 핵심 예외도 잘 처리하고 있어서 큰 문제는 없다.  
오히려 이 문제에서 자주 틀리는 건 예외 처리를 빼먹는 경우다.

예를 들어 아래처럼 단순하게 생각하면

```python
floor = n % h
room = n // h + 1
```

`n`이 `h`의 배수일 때 문제가 생긴다.

```text
H = 6, N = 12
```

이 경우 실제 답은 `602`여야 한다.  
그런데 위 공식을 그대로 쓰면

- `floor = 12 % 6 = 0`
- `room = 12 // 6 + 1 = 3`

이 되어 `003` 같은 이상한 결과 방향으로 가버린다.

내 코드는 이 상황을

```python
if n % h == 0:
```

으로 따로 처리해서 잘 막고 있다.  
즉, 이 문제의 핵심은 공식 자체보다 "배수일 때 예외"를 놓치지 않는 것이다.

그리고 한 가지 더 재미있는 포인트는 `w`가 계산식에 직접 쓰이지 않는다는 점이다.  
처음에는 이상해 보이지만, `W`는 호텔이 실제로 수용 가능한 범위를 정해 주는 역할만 한다.  
문제에서 이미 `1 ≤ N ≤ H × W`를 보장하므로, 몇 번째 손님이 몇 호 라인에 들어가는지는 `H`와 `N`만으로도 결정된다.

## 개선 코드 1

내 원래 코드도 충분히 좋지만, 방 번호를 문자열 포맷으로 표현하면 조금 더 읽기 쉬울 수 있다.

```python
import sys

input = sys.stdin.readline

t = int(input())

for _ in range(t):
    h, w, n = map(int, input().split())

    floor = n % h
    room = n // h + 1

    if floor == 0:  # h의 배수라면 꼭대기 층이면서, 호수는 하나 줄어든다
        floor = h
        room -= 1

    print(f"{floor}{room:02d}")
```

여기서 `f"{floor}{room:02d}"`의 `:02d`는 "정수를 두 자리로 출력하되, 한 자리면 앞을 0으로 채워라"는 뜻이다.  
예를 들어 `room = 3`이면 `03`으로 출력된다.  
그래서 `12층 3호`는 `1203`처럼 문제 형식에 맞게 출력할 수 있다.

## 개선 코드 2

조금 더 깔끔하게 쓰려면 아예 `N - 1` 기준의 0번 인덱스로 바꿔서 예외 분기를 없앨 수도 있다.

```python
import sys

input = sys.stdin.readline

t = int(input())

for _ in range(t):
    h, w, n = map(int, input().split())

    # 1번째 손님을 0번째로 바꿔 생각하면 배수 예외를 따로 처리하지 않아도 된다
    floor = (n - 1) % h + 1
    room = (n - 1) // h + 1

    print(f"{floor}{room:02d}")
```

이 방식은 `n % h == 0` 같은 조건문이 없어도 된다.  
처음부터 "손님 번호를 0부터 시작한다고 생각하면" 마지막 층 문제도 자연스럽게 해결된다.

개인적으로는 설명할 때는 원래 코드가 더 직관적이고, 다시 작성할 때는 이 버전이 더 깔끔하다고 느꼈다.

## 시간 복잡도

- 원래 코드: 테스트 케이스 하나당 `O(1)`
- 개선 코드 1: 테스트 케이스 하나당 `O(1)`
- 개선 코드 2: 테스트 케이스 하나당 `O(1)`

여기서 `T`는 테스트 케이스 개수다.  
각 테스트 케이스에서는 나머지 연산, 몫 연산, 덧셈, 조건문 같은 고정 개수의 연산만 수행한다.

즉, `H`, `W`, `N`의 숫자가 커지더라도 반복문이 더 늘어나지 않는다.  
손님을 1명부터 `N`명까지 직접 시뮬레이션하지 않고, 규칙을 식으로 한 번에 계산하기 때문이다.

그래서 전체 입력 기준으로 보면 총 시간 복잡도는 `O(T)`라고 볼 수 있다.  
테스트 케이스 수가 2배가 되면 계산 횟수도 대체로 2배가 되지만, 각 케이스 안에서는 늘어나는 작업이 거의 없다.

반대로 손님을 실제로 한 명씩 배정하는 방식으로 풀면 테스트 케이스 하나당 `O(N)`까지 갈 수 있다.  
이 문제에서는 반복되는 배정 과정을 수식으로 압축했기 때문에 훨씬 간단하고 빠르다.

## 이번 문제에서 배운 점

- 겉으로는 구현 문제처럼 보여도, 실제로는 배정 순서의 규칙을 찾는 문제가 더 가깝다.
- `n % h == 0`처럼 배수일 때 생기는 예외는 수학 문제에서 자주 나오는 함정이다.
- `N - 1` 기준으로 바꾸면 예외 분기를 없앨 수 있어서 코드가 더 매끈해질 수 있다.
