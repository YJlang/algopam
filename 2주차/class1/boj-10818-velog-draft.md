# [알고팜 2주차] BOJ 10818 최소, 최대 - `min()`과 `max()`도 결국 끝까지 본다

이번 문제는 Python 내장 함수 `min()`과 `max()`를 쓰면 아주 짧게 풀 수 있었다. 처음 풀이도 정답 방향은 맞았지만, 다시 보니 "내장 함수가 알아서 해준다"에서 멈추면 시간 복잡도가 왜 `O(n)`인지 흐릿하게 남을 것 같았다. 그래서 이번에는 `min()`, `max()`가 실제로 어떤 일을 하는지와 최악의 경우까지 같이 정리해봤다.

문제 링크: https://www.acmicpc.net/problem/10818

## 문제 정보

- 시간 제한: 1초
- 메모리 제한: 256MB
- 풀이 감각: 구현, 배열, 최솟값/최댓값 찾기

## 문제 설명

문제 원문은 위 링크에서 확인할 수 있고, 아래는 풀이를 위해 다시 정리한 내용이다.

- 정수 `N`개가 주어진다.
- 주어진 정수들 중에서 가장 작은 값과 가장 큰 값을 찾아야 한다.
- 두 값을 한 줄에 공백으로 구분해서 출력하면 된다.
- 정수의 개수는 최대 1,000,000개까지 들어올 수 있다.

## 입력 형식

- 첫째 줄에 정수의 개수 `N`이 주어진다.
- 둘째 줄에 `N`개의 정수가 공백으로 구분되어 주어진다.
- 각 정수는 `-1,000,000` 이상 `1,000,000` 이하이다.

## 출력 형식

- 첫째 줄에 주어진 정수들 중 최솟값과 최댓값을 공백으로 구분해 출력한다.

## 예제 입력 1

```text
5
20 10 35 30 7
```

## 예제 출력 1

```text
7 35
```

## 접근 아이디어

1. 둘째 줄에 들어온 숫자들을 정수 리스트로 바꾼다.
2. 리스트 전체에서 가장 작은 값은 `min()`으로 찾는다.
3. 리스트 전체에서 가장 큰 값은 `max()`로 찾는다.

`min(num_list)`는 `num_list` 안의 값을 앞에서부터 확인하면서 가장 작은 값을 반환한다. `max(num_list)`도 같은 방식으로 가장 큰 값을 반환한다. 즉, 이름만 보면 한 번에 답이 나오는 것 같지만 내부적으로는 리스트를 끝까지 확인해야 한다.

예를 들어 `[20, 10, 35, 30, 7]`에서 `min()`은 모든 값을 보면서 지금까지 본 값 중 가장 작은 값을 갱신하고, 마지막에 `7`을 반환한다. `max()`도 모든 값을 확인한 뒤 `35`를 반환한다.

## 내가 처음 작성한 코드

아래 코드는 블로그에서 읽기 쉽도록 필요한 설명 주석을 덧붙여 정리한 버전이다.

```python
# BOJ 10818: 최소, 최대
# https://www.acmicpc.net/problem/10818
# 브론즈3
import sys

input = sys.stdin.readline

n = int(input())  # 정수의 개수
num_list = list(map(int, input().split()))  # 둘째 줄의 숫자들을 정수 리스트로 변환

# min()과 max()는 각각 리스트 전체를 한 번씩 확인한다.
print(min(num_list), max(num_list))
```

이 코드는 문제 조건에 맞게 입력이 들어온다는 전제에서는 잘 동작한다. 실제 예제 입력을 넣으면 `7 35`가 출력된다.

## 이 코드에서 내가 하려고 했던 것

처음 풀이에서는 입력을 모두 리스트로 만든 다음, Python이 제공하는 내장 함수에 최솟값과 최댓값 찾기를 맡겼다.

여기서 `map(int, input().split())`은 입력 문자열을 공백 기준으로 나눈 뒤 각각을 `int`로 바꿔준다. 그리고 `list(...)`로 감싸면 그 결과를 한 번에 저장한 리스트가 된다.

이 문제에서는 숫자가 모두 한 줄에 주어지므로, `input().split()`으로 한 줄을 쪼개고 `map(int, ...)`로 정수로 바꾸는 방식이 자연스럽다.

## 아쉬웠던 점

정답 자체는 맞지만, 두 가지 정도는 짚고 넘어갈 만하다.

첫째, `n`을 입력받아 놓고 실제로는 사용하지 않는다. BOJ 입력은 문제 조건에 맞게 들어오므로 틀린 풀이는 아니지만, 코드를 읽는 입장에서는 "`N`개라는 조건을 확인하지 않아도 괜찮나?"라는 생각이 들 수 있다.

둘째, `min(num_list)`와 `max(num_list)`를 따로 호출하기 때문에 리스트를 두 번 훑는다. Big-O로 보면 여전히 `O(n)`이지만, 실제로는 최솟값을 찾기 위해 한 번, 최댓값을 찾기 위해 한 번 확인하는 구조다.

```text
틀리는 반례라기보다는 개선 포인트:
N이 1,000,000처럼 커지면 min()과 max()가 각각 1,000,000개 값을 확인한다.
즉, 답은 맞지만 같은 리스트를 두 번 보는 셈이다.
```

## 개선 풀이 1

아래 코드는 변수와 핵심 동작이 눈에 들어오도록 짧은 주석을 덧붙인 버전이다.

```python
import sys

input = sys.stdin.readline

_ = int(input())  # 개수 정보는 입력 형식을 맞추기 위해 읽기만 한다.
numbers = list(map(int, input().split()))

print(min(numbers), max(numbers))
```

이 버전은 원래 풀이와 거의 같은 방식이다. 다만 실제로 사용하지 않는 값은 관례적으로 `_`에 담아서 "입력은 받지만 이후 계산에는 쓰지 않는다"는 의도를 조금 더 분명하게 했다.

`min()`과 `max()`를 쓰는 풀이가 이 문제에서는 가장 간단하고 읽기 쉽다. 특히 Bronze 단계에서는 내장 함수가 어떤 값을 반환하는지 정확히 알고 쓰는 것도 좋은 풀이 방법이라고 생각한다.

## 개선 풀이 2

아래 코드는 블로그 독자가 흐름을 따라가기 쉽도록 설명 주석을 덧붙인 버전이다.

```python
import sys

input = sys.stdin.readline

n = int(input())
numbers = list(map(int, input().split()))

# 첫 번째 값을 기준으로 시작하면 음수만 있는 경우에도 안전하다.
min_value = numbers[0]
max_value = numbers[0]

for x in numbers:
    # 더 작은 값을 만나면 최솟값 후보를 바꾼다.
    if x < min_value:
        min_value = x

    # 더 큰 값을 만나면 최댓값 후보를 바꾼다.
    if x > max_value:
        max_value = x

print(min_value, max_value)
```

이 버전은 `min()`과 `max()`를 따로 부르지 않고, 리스트를 한 번만 돌면서 최솟값과 최댓값을 동시에 갱신한다.

주의할 점은 `min_value`와 `max_value`를 `0` 같은 임의의 값으로 시작하면 안 된다는 것이다. 입력이 모두 음수일 수도 있고, 모두 양수일 수도 있기 때문이다. 그래서 실제 입력에 들어 있는 첫 번째 숫자로 시작하는 편이 안전하다.

예를 들어 숫자가 `[-5, -3, -10]`이라면 최댓값은 `-3`이다. 만약 `max_value = 0`으로 시작했다면 입력에 없는 `0`을 최댓값처럼 들고 가게 되어 틀린 답이 나온다.

## 시간 복잡도

- 내가 처음 작성한 풀이: `O(n)`
- 개선 풀이 1: `O(n)`
- 개선 풀이 2: `O(n)`

여기서 `n`은 입력으로 주어지는 정수의 개수다. 이 문제에서는 `N`이 최대 1,000,000까지 될 수 있다.

최악의 경우는 `n`이 가장 큰 경우, 즉 정수가 1,000,000개 들어오는 경우로 생각하면 된다. 값의 순서가 오름차순인지 내림차순인지는 큰 차이가 없다. 최솟값과 최댓값을 정확히 알려면 결국 모든 값을 한 번은 확인해야 하기 때문이다.

내가 처음 작성한 풀이는 `list(map(...))`로 숫자 `n`개를 리스트에 담을 때 한 번 `O(n)`이 걸린다. 그다음 `min(num_list)`가 리스트 전체를 한 번 보면서 `O(n)`, `max(num_list)`도 리스트 전체를 한 번 보면서 `O(n)`이 걸린다.

그래서 실제 흐름으로는 대략 `O(n) + O(n) + O(n)`처럼 볼 수 있고, Big-O에서는 상수 배를 생략하므로 전체 시간 복잡도는 `O(n)`이다.

개선 풀이 1도 같은 방식으로 리스트를 만들고 `min()`, `max()`를 호출하므로 `O(n)`이다. 원래 풀이보다 의도는 조금 더 명확하지만, 시간 복잡도가 더 좋아지는 풀이는 아니다.

개선 풀이 2는 리스트를 만든 뒤 `for x in numbers`로 한 번만 순회하면서 최솟값과 최댓값을 같이 갱신한다. 그래서 리스트를 만드는 비용 `O(n)`과 순회 비용 `O(n)`을 합쳐 전체적으로 `O(n)`이다.

조금 더 쉽게 말하면, 입력 숫자가 10개에서 20개로 늘어나면 확인해야 할 숫자 수도 거의 2배가 된다. 1,000개에서 2,000개로 늘어나도 마찬가지다. 그래서 이 문제의 핵심 작업은 입력 크기에 비례해서 늘어나는 `O(n)`이라고 볼 수 있다.

다만 같은 `O(n)`이어도 작업량에는 차이가 있다. 처음 풀이와 개선 풀이 1은 `min()`과 `max()`가 리스트를 각각 확인하므로 값을 찾는 과정만 보면 두 번 순회한다. 개선 풀이 2는 최솟값과 최댓값을 한 반복문 안에서 같이 갱신하므로 개념적으로는 한 번 순회한다. 대신 Python에서는 내장 함수가 C로 구현되어 있어 실제 실행 시간은 직접 짠 `for`문보다 내장 함수 풀이가 더 빠를 수도 있다.

## 이번 문제에서 배운 점

- `min()`과 `max()`는 편하지만, 각각 리스트 전체를 확인하는 함수다.
- Big-O에서 `O(n) + O(n)`은 결국 `O(n)`으로 표현하지만, 실제 반복 횟수 차이는 설명해두는 게 좋다.
- 최솟값과 최댓값을 직접 갱신할 때는 `0` 같은 임의의 값보다 실제 첫 번째 입력값으로 초기화하는 편이 안전하다.
