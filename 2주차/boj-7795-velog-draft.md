# [알고팜 2주차] BOJ 7795 먹을 것인가 먹힐 것인가 - `bisect_left`가 왜 정답 개수가 되는지 이해하기

이번 문제는 처음 보면 "A의 각 원소마다 B를 다 비교해야 하나?" 싶어서 완전탐색부터 떠올리기 쉽다. 그런데 입력 크기를 보면 그렇게 풀기에는 부담이 있고, 정렬과 `bisect_left`를 이용하면 훨씬 빠르게 풀 수 있다. 이번 글에서는 내가 작성한 코드가 왜 맞는지, 특히 `bisect_left(b, x)`가 왜 "x보다 작은 B의 개수"가 되는지를 초보자 눈높이에서 다시 정리해보려고 한다.

문제 링크: https://www.acmicpc.net/problem/7795

## 문제 정보

- 시간 제한: 1초
- 메모리 제한: 256MB
- 풀이 감각: 정렬, 이분탐색, 쌍의 개수 세기

## 문제 설명

문제 원문은 위 링크에서 확인할 수 있고, 아래는 풀이를 위해 다시 정리한 내용이다.

- 심해 생명체 A와 B가 있다.
- A는 자기보다 크기가 작은 B만 먹을 수 있다.
- 즉, `A > B`인 쌍의 개수를 세면 된다.
- 각 테스트케이스마다 그런 쌍이 몇 개인지 출력해야 한다.

## 입력 형식

- 첫째 줄에 테스트케이스 개수 `T`가 주어진다.
- 각 테스트케이스의 첫째 줄에 `N`, `M`이 주어진다.
- 둘째 줄에는 A의 크기들, 셋째 줄에는 B의 크기들이 주어진다.

## 출력 형식

- 각 테스트케이스마다 `A > B`인 쌍의 개수를 한 줄에 하나씩 출력한다.

## 예제 입력 1

```text
2
5 3
8 1 7 3 1
3 6 1
3 4
2 13 7
103 11 290 215
```

## 예제 출력 1

```text
7
1
```

## 접근 아이디어

1. `A`의 각 원소 `x`에 대해, `B`에서 `x`보다 작은 원소가 몇 개인지만 알면 된다.
2. `B`를 정렬하면 "`x`보다 작은 값들"과 "`x` 이상인 값들" 사이에 경계가 생긴다.
3. 그 경계를 `bisect_left`로 찾으면, 그 위치 자체가 곧 `x`보다 작은 원소의 개수가 된다.

## `bisect_left`가 여기서 하는 일

`bisect_left(b, x)`는 정렬된 리스트 `b`에 `x`를 넣는다면 **가장 왼쪽 몇 번째 위치에 들어가야 하는지**를 반환한다.

예를 들어,

```python
b = [1, 3, 6]
x = 7
```

이라면 `7`은 `6` 뒤에 들어가야 하므로 위치는 `3`이다.

```text
[1, 3, 6]
         ^
      index 3
```

이 `3`은 곧 `7`보다 작은 값이 3개 있다는 뜻과 같다. 그래서 `result += bisect_left(b, x)`를 하면 "현재 `x`가 먹을 수 있는 B의 개수"를 더하는 것과 같은 의미가 된다.

또 하나 중요한 점은 이 문제 조건이 `A > B`라는 것이다. 같은 크기는 먹을 수 없으므로, `bisect_right`가 아니라 `bisect_left`가 더 잘 맞는다.

예를 들어,

```python
b = [1, 3, 3, 5]
x = 3
```

일 때 `3`보다 작은 값은 `1` 하나뿐이다. 이때 `bisect_left(b, 3)`은 `1`을 반환하므로 정확하게 원하는 값을 돌려준다.

## 내가 처음 작성한 코드

아래 코드는 블로그에서 읽기 쉽도록 필요한 설명 주석을 덧붙여 정리한 버전이다.

```python
import sys
from bisect import bisect_left  # 정렬된 리스트에서 삽입 위치를 찾는 함수

input = sys.stdin.readline

t = int(input())  # 테스트케이스 개수

for _ in range(t):
    n, m = map(int, input().split())  # A의 개수, B의 개수
    a = list(map(int, input().split()))
    b = list(map(int, input().split()))

    # bisect_left를 쓰려면 탐색 대상이 정렬되어 있어야 한다.
    b.sort()

    result = 0

    for x in a:
        # x가 들어갈 가장 왼쪽 위치 = x보다 작은 B 원소의 개수
        result += bisect_left(b, x)

    print(result)
```

이 코드의 핵심은 "`A`를 하나씩 보면서, 각 `x`보다 작은 `B`가 몇 개인지 빠르게 더한다"는 흐름이다. 겉보기엔 짧지만, `bisect_left`가 무엇을 돌려주는지 이해하면 꽤 깔끔한 풀이라는 걸 알 수 있다.

## 이 코드에서 내가 하려고 했던 것

내가 하려던 것은 각 `A` 원소마다 "`B`를 전부 보지 않고도" 먹을 수 있는 먹이 수를 빠르게 구하는 것이었다.

완전탐색으로 풀면 `A`의 각 원소마다 `B`를 모두 비교해야 하니까 시간이 많이 든다. 그래서 `B`만 정렬해두고, 각 `x`에 대해 "작은 값이 몇 개 있는지"를 이분탐색으로 바로 구하려고 한 코드였다.

즉, "쌍을 하나하나 직접 세지 말고, 각 `x`가 먹을 수 있는 개수를 한 번에 구해서 더하자"는 방향이었다.

## 아쉬웠던 점

이 코드는 정답 코드이지만, 초보자 입장에서는 두 군데에서 멈추기 쉽다.

첫째, `bisect_left`가 단순히 "이분탐색 함수"라고만 적혀 있으면 실제로 무슨 값을 돌려주는지 바로 감이 오지 않는다. 이 문제에서는 "삽입 위치"가 중요한 게 아니라, 그 삽입 위치가 곧 "`x`보다 작은 원소 수"라는 점을 이해해야 한다.

둘째, `b.sort()`를 왜 하는지 놓치면 안 된다. 이분탐색은 정렬된 배열에서만 가능하다. 정렬이 되어 있어야 "`x`보다 작은 값들"과 "`x` 이상인 값들"이 한 경계로 나뉘고, 그 경계를 반씩 줄여가며 찾을 수 있다.

```text
이 코드의 핵심은 "bisect_left를 쓴다"가 아니라
"정렬된 B에서 x보다 작은 값의 개수를 삽입 위치로 바꿔서 센다"는 발상이다.
```

## 개선 풀이 1

아래 코드는 변수 이름과 주석을 조금 더 풀어서, 처음 읽는 사람도 흐름을 따라가기 쉽게 정리한 버전이다.

```python
import sys
from bisect import bisect_left  # 정렬된 리스트에서 왼쪽 삽입 위치를 찾는다

input = sys.stdin.readline

test_case_count = int(input())

for _ in range(test_case_count):
    a_count, b_count = map(int, input().split())
    a_sizes = list(map(int, input().split()))
    b_sizes = list(map(int, input().split()))

    # B를 정렬해야 이분탐색을 할 수 있다.
    b_sizes.sort()

    answer = 0

    for a_size in a_sizes:
        # a_size가 들어갈 왼쪽 위치를 찾는다.
        # 그 위치보다 왼쪽에는 모두 a_size보다 작은 값만 있다.
        smaller_b_count = bisect_left(b_sizes, a_size)
        answer += smaller_b_count

    print(answer)
```

풀이 자체는 같지만, 변수 이름을 `a_count`, `b_count`, `a_size`, `smaller_b_count`처럼 바꾸면 역할이 훨씬 바로 보인다. 회고 글에서는 이런 식으로 이름을 풀어 쓰는 것만으로도 읽는 사람이 이해하기 쉬워진다.

## 개선 풀이 2

아래 코드는 `bisect_left` 대신 투 포인터처럼 생각하는 방법이다.

```python
import sys

input = sys.stdin.readline

t = int(input())

for _ in range(t):
    n, m = map(int, input().split())
    a = list(map(int, input().split()))
    b = list(map(int, input().split()))

    # 두 배열을 모두 정렬해 두면, 작은 값 개수를 누적해서 셀 수 있다.
    a.sort()
    b.sort()

    answer = 0
    b_index = 0  # 현재까지 "A보다 작은 B"의 개수를 가리키는 포인터

    for a_value in a:
        # b[b_index]가 아직 a_value보다 작다면 먹을 수 있는 후보이므로 포인터를 전진
        while b_index < m and b[b_index] < a_value:
            b_index += 1

        # 지금 b_index는 곧 a_value보다 작은 B의 개수
        answer += b_index

    print(answer)
```

이 방법은 `A`도 정렬해서 앞에서부터 보면서, `B` 쪽 포인터를 한 번만 전진시키는 방식이다. `A` 값이 커질수록 "`그보다 작은 B의 개수`"도 줄어들지 않는다는 점을 이용한다.

개인적으로는 `bisect_left` 풀이가 더 짧고 문제 포인트가 잘 드러난다고 느꼈지만, 투 포인터 풀이는 "왜 이전에 센 개수를 다시 세지 않아도 되는지"를 연습하기 좋다.

## 시간 복잡도

- 내가 처음 작성한 풀이: 테스트케이스 하나당 `O(m log m + n log m)`
- 개선 풀이 1: 테스트케이스 하나당 `O(m log m + n log m)`
- 개선 풀이 2: 테스트케이스 하나당 `O(n log n + m log m + n + m)`

여기서

- `n`은 A의 원소 개수
- `m`은 B의 원소 개수

를 뜻한다.

처음 풀이와 개선 풀이 1은 구조가 같다.

- `b.sort()`는 `m`개를 정렬하므로 `O(m log m)`
- `a`를 한 번 순회하는 데 `O(n)`
- 각 `a` 원소마다 `bisect_left`를 한 번 호출하는데, 이 함수는 정렬된 `b`에서 탐색 범위를 절반씩 줄여가므로 `O(log m)`

그래서 전체는

```text
O(m log m + n log m)
```

이 된다.

초보자 식으로 풀어 말하면 이렇다.

- `B`를 먼저 한 번 정렬하는 비용이 있다.
- 그 다음 `A`를 하나씩 보는데, 매번 `B` 전체를 다 보는 게 아니라 절반, 절반, 절반으로 줄여가며 찾는다.
- 그래서 `A` 원소가 `n`개 있으면, "빠른 탐색"을 `n`번 하는 구조가 된다.

`O(log m)`이 낯설다면 "탐색 범위를 매번 반으로 줄이는 비용"이라고 생각하면 된다.

예를 들어 `m = 20000`이면,

- 한 칸씩 찾는 선형탐색은 최악 20000번 가까이 볼 수 있지만
- 이분탐색은 대략 14~15번 정도만 비교하면 범위를 거의 다 좁힐 수 있다

그래서 큰 입력에서 차이가 크게 난다.

개선 풀이 2는

- `a.sort()`에 `O(n log n)`
- `b.sort()`에 `O(m log m)`
- 이후 포인터 두 개는 전체를 한 번씩만 지나가므로 `O(n + m)`

이 걸린다.

즉, 두 번째 풀이는 "정렬 비용 + 한 번 훑는 비용"이고, 첫 번째 풀이는 "정렬 비용 + `n`번의 이분탐색 비용"이라고 볼 수 있다.

둘 다 충분히 빠르지만, 이번 문제에서는 `bisect_left` 풀이가 짧고 구현 실수도 적어서 꽤 좋은 선택이라고 느꼈다.

## 이번 문제에서 배운 점

- `A > B` 쌍의 개수를 세는 문제는 "A마다 먹을 수 있는 B 개수"로 바꿔 생각하면 훨씬 단순해진다.
- `bisect_left`는 단순히 이분탐색 함수가 아니라, 이 문제에서는 "`x`보다 작은 값의 개수"를 바로 알려주는 도구로 볼 수 있다.
- 시간 복잡도는 식만 외우기보다, "무엇을 몇 번 반복하는지"를 같이 봐야 진짜 이해가 된다.
