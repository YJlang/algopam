# [알고리즘 3주차] BOJ 20310 타노스 - 지우는 방향이 답을 바꾼다

예제는 맞았는데 제출하면 틀리는 문제였다. 처음에는 0과 1을 절반씩 지우면 된다고 생각했는데, 이 문제의 핵심은 단순히 개수를 맞추는 것이 아니라 **원래 문자열의 순서를 유지한 채 가능한 문자열 중 사전순으로 가장 빠른 것**을 만드는 데 있었다.

문제 링크: https://www.acmicpc.net/problem/20310

## 문제 정보

- 시간 제한: 1초
- 메모리 제한: 1024MB
- 분류 감각: 문자열, 그리디

## 문제 설명

0과 1로만 이루어진 문자열 `S`가 주어진다.

문자열 안에 있는 0의 개수와 1의 개수는 각각 짝수이다. 타노스는 이 문자열에서 0의 절반, 1의 절반을 제거해서 새로운 문자열 `S'`를 만들려고 한다.

이때 문자를 제거할 수는 있지만, 남은 문자들의 상대적인 순서는 바꿀 수 없다. 가능한 `S'` 중에서 사전순으로 가장 앞서는 문자열을 출력해야 한다.

## 입력 형식

- 첫째 줄에 문자열 `S`가 주어진다.
- `S`의 길이는 2 이상 500 이하이다.
- `S`는 짝수 개의 0과 짝수 개의 1로 이루어져 있다.

## 출력 형식

- 0의 절반과 1의 절반을 제거해서 만들 수 있는 문자열 중 사전순으로 가장 빠른 문자열을 출력한다.

## 예제 입력 1

```text
1010
```

## 예제 출력 1

```text
01
```

## 예제 입력 2

```text
000011
```

## 예제 출력 2

```text
001
```

## 접근 아이디어

사전순으로 작은 문자열을 만들려면 앞쪽에 가능한 한 `0`이 많이 남아야 한다. `0`은 `1`보다 사전순으로 작기 때문이다.

그래서 제거 방향을 이렇게 잡아야 한다.

1. `1`은 앞에서부터 제거한다.
2. `0`은 뒤에서부터 제거한다.
3. 제거 표시가 안 된 문자만 원래 순서대로 출력한다.

처음에는 이 방향을 반대로 생각했다. 그런데 `0`을 앞에서 지워버리면 사전순으로 유리한 문자를 스스로 버리는 셈이고, `1`을 뒤에서 지우면 앞쪽의 큰 문자가 그대로 남는다.

## 원래 작성했던 코드

아래는 처음에 생각했던 방향을 블로그용으로 다시 정리한 코드다. 예제 일부는 맞을 수 있지만, 제거 방향이 반대라서 항상 사전순 최소를 만들지는 못한다.

```python
import sys
input = sys.stdin.readline

S = input().strip()

remove_zero = S.count('0') // 2
remove_one = S.count('1') // 2

removed = [False] * len(S)

# 잘못된 방향: 0을 앞에서부터 지우면 앞쪽의 작은 문자를 잃는다.
cnt = 0
for i in range(len(S)):
    if S[i] == '0' and cnt < remove_zero:
        removed[i] = True
        cnt += 1

# 잘못된 방향: 1을 뒤에서부터 지우면 앞쪽의 큰 문자가 남는다.
cnt = 0
for i in range(len(S) - 1, -1, -1):
    if S[i] == '1' and cnt < remove_one:
        removed[i] = True
        cnt += 1

print(''.join(S[i] for i in range(len(S)) if not removed[i]))
```

## 이 코드에서 하려던 것

이 코드는 0과 1의 개수를 각각 센 다음, 정확히 절반씩 제거하려고 했다. `removed` 배열을 만들어서 실제 문자열을 바로 지우지 않고, 지울 위치만 표시한 뒤 마지막에 표시되지 않은 문자만 출력한다.

이 방식 자체는 괜찮다. 문자열 중간에서 `pop`이나 `del`로 계속 삭제하면 인덱스가 밀려서 헷갈릴 수 있는데, `removed`로 표시하면 원래 위치를 유지하면서 생각할 수 있다.

문제는 **어느 쪽에서 지우느냐**였다.

## 약했던 점과 반례

반례는 예제 1번 그대로도 확인할 수 있다.

```text
입력
1010
```

처음 코드처럼 `0`을 앞에서 지우고 `1`을 뒤에서 지우면 남는 문자열은 다음과 같다.

```text
10
```

하지만 가능한 문자열 중 사전순으로 가장 작은 답은 다음과 같다.

```text
01
```

`0`이 앞에 오는 `01`이 `10`보다 사전순으로 더 작다. 결국 이 문제는 “몇 개를 지울지”보다 “어디에서 지울지”가 더 중요했다.

## 개선 풀이 1: 제거 표시 배열 사용

현재 파일의 풀이 방향이다. 앞쪽의 `1`을 먼저 제거하고, 뒤쪽의 `0`을 제거한다.

```python
import sys
input = sys.stdin.readline

S = input().strip()

# 각각 절반만 제거해야 한다.
remove_zero = S.count('0') // 2
remove_one = S.count('1') // 2

# True인 위치는 마지막 출력에서 제외한다.
removed = [False] * len(S)

# 1은 0보다 크므로 앞에서부터 지워야 사전순으로 작아진다.
cnt = 0
for i in range(len(S)):
    if S[i] == '1' and cnt < remove_one:
        removed[i] = True
        cnt += 1

# 0은 앞쪽에 남길수록 좋으므로 뒤에서부터 지운다.
cnt = 0
for i in range(len(S) - 1, -1, -1):
    if S[i] == '0' and cnt < remove_zero:
        removed[i] = True
        cnt += 1

# 지워지지 않은 문자만 원래 순서대로 이어 붙인다.
print(''.join(S[i] for i in range(len(S)) if not removed[i]))
```

핵심은 `0 < 1`이라는 점이다. 앞쪽의 `1`을 지우면 더 작은 문자가 앞으로 올 기회가 생긴다. 반대로 `0`은 앞쪽에 있을수록 유리하므로 뒤쪽의 `0`부터 제거한다.

`removed` 배열은 각 문자를 실제로 삭제하지 않고 “이 위치는 지운 것으로 치자”라고 표시하는 용도다. 그래서 반복 중에 문자열 길이가 바뀌지 않고, 인덱스가 밀리는 문제도 없다.

## 개선 풀이 2: 한 번 훑으면서 답 만들기

같은 생각을 조금 더 직접적으로 쓰면, 답 문자열을 만들면서 처리할 수도 있다.

```python
import sys
input = sys.stdin.readline

S = input().strip()

keep_zero = S.count('0') // 2      # 앞에서부터 남길 0의 개수
remove_one = S.count('1') // 2     # 앞에서부터 버릴 1의 개수

answer = []

for ch in S:
    if ch == '0':
        # 0은 앞쪽 것을 남기는 게 유리하다.
        if keep_zero > 0:
            answer.append(ch)
            keep_zero -= 1
    else:
        # 1은 앞쪽 것을 먼저 버리고, 남은 1만 답에 넣는다.
        if remove_one > 0:
            remove_one -= 1
        else:
            answer.append(ch)

print(''.join(answer))
```

이 버전은 “앞쪽 0은 살리고, 앞쪽 1은 버린다”는 규칙이 더 잘 보인다. `answer`는 최종으로 출력할 문자를 담는 리스트다. 파이썬에서 문자열에 계속 `+`로 붙이면 매번 새 문자열을 만들 수 있으므로, 여러 글자를 모을 때는 리스트에 `append`한 뒤 마지막에 `join`하는 편이 깔끔하다.

## 시간 복잡도

여기서 `n`은 문자열 `S`의 길이이다.

- 처음 잘못된 풀이: `O(n)`
- 개선 풀이 1: `O(n)`
- 개선 풀이 2: `O(n)`

처음 풀이와 개선 풀이 1은 `S.count('0')`, `S.count('1')`에서 문자열을 각각 한 번씩 훑는다. 이후 앞에서 한 번, 뒤에서 한 번, 마지막 출력에서 한 번 더 훑는다. 여러 번 훑기는 하지만 각 반복은 최대 `n`번 정도만 실행되므로 전체 시간 복잡도는 `O(n)`이다.

개선 풀이 2도 개수를 세기 위해 `count`를 사용하고, 답을 만들기 위해 문자열을 한 번 더 훑는다. 마찬가지로 반복 횟수는 입력 길이에 비례하므로 `O(n)`이다.

입력 길이가 대략 두 배가 되면 확인해야 하는 문자 수도 대략 두 배가 된다. 중첩 반복문으로 모든 문자 쌍을 비교하는 구조가 아니기 때문에 `O(n^2)`처럼 급격히 늘어나지는 않는다.

공간 복잡도도 `O(n)`이다. 개선 풀이 1은 `removed` 배열을 만들고, 개선 풀이 2는 `answer` 리스트를 만들기 때문이다. 둘 다 최악의 경우 문자열 길이에 비례하는 추가 공간을 사용한다.

## 이번 문제에서 배운 점

- 예제가 맞아도 “가능한 문자열 중”이라는 조건을 놓치면 반례가 바로 생긴다.
- 사전순 최소를 만들 때는 앞쪽 문자가 특히 중요하다.
- 이 문제에서는 `1`은 앞에서 지우고, `0`은 뒤에서 지우는 방향이 그리디의 핵심이다.
- 문자열을 직접 삭제하는 대신 삭제 표시 배열이나 답 리스트를 쓰면 인덱스가 꼬일 위험이 줄어든다.
