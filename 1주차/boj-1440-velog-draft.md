알고팜 1주차에서 풀었던 **BOJ 1440 - 타임머신**을 정리해보려고 한다.
이 문제는 구현 문제답게 아이디어가 아주 크지는 않지만, 조건을 정확히 읽고 경우의 수를 빠짐없이 세는 감각이 필요한 문제였다.
이번 글에서는 내가 처음 작성한 코드가 어떤 방식으로 정답을 세고 있는지, 그 방식이 왜 맞는지, 그리고 조금 더 정리된 형태로는 어떻게 쓸 수 있는지 같이 남겨보겠다.

문제 링크: https://www.acmicpc.net/problem/1440

## 문제 정보

- 시간 제한: 2초
- 메모리 제한: 128MB
- 풀이 감각: 구현 / 브루트포스

## 문제 설명

문제 원문은 위 링크에서 확인할 수 있고, 아래는 풀이를 위해 다시 정리한 내용이다.

- 디지털 시계에 `DD:DD:DD` 형태의 숫자가 주어진다.
- 이 세 칸이 실제로 `시:분:초` 순서인지, 아니면 다른 순서인지 알 수 없다.
- 시간은 `01~12`, 분과 초는 각각 `00~59` 범위만 허용된다.
- 세 숫자를 시, 분, 초에 배치하는 모든 방법 중에서 유효한 해석의 개수를 구하면 된다.
- 숫자 값이 같더라도, 시/분/초로 읽는 위치가 다르면 다른 방법으로 센다.

핵심은 결국 `세 칸을 시, 분, 초로 배치하는 경우를 모두 확인하고`, 그중에서 범위 조건을 만족하는 경우만 세는 것이다.

## 입력 형식

- 첫째 줄에 시간이 `DD:DD:DD` 형태로 주어진다.
- 문자열 길이는 항상 8이고, 각 `D`는 `0~9` 사이 숫자이다.

## 출력 형식

- 주어진 숫자를 유효한 시각으로 읽을 수 있는 방법의 수를 출력한다.
- 가능한 방법이 하나도 없으면 `0`을 출력한다.

## 예제 입력 1

```text
00:00:00
```

## 예제 출력 1

```text
0
```

## 예제 입력 2

```text
01:00:00
```

## 예제 출력 2

```text
2
```

## 예제 입력 3

```text
01:12:23
```

## 예제 출력 3

```text
4
```

## 접근 아이디어

이 문제에서 확인해야 할 것은 많지 않다.

1. 입력된 세 숫자를 각각 하나의 칸으로 본다.
2. 그 세 칸을 `시, 분, 초`에 배치하는 경우는 총 6가지다.
3. 각 경우마다 `시는 1~12`, `분과 초는 0~59`인지 검사한다.
4. 조건을 만족할 때마다 정답을 1씩 증가시키면 된다.

즉, 입력 크기가 커지는 문제가 아니라 **정해진 6가지 배치를 검사하는 구현 문제**라고 볼 수 있다.

## 내가 처음 작성한 코드

```python
#1주차 과제 BOJ1440
#https://www.acmicpc.net/problem/1440
import sys

input = sys.stdin.readline

time = list(map(int, input().split(':')))

h,m,s = time[0],time[1],time[2]

cnt = 0

lists = [
    (h,m,s),
    (h,s,m),
    (m,h,s),
    (m,s,h),
    (s,h,m),
    (s,m,h)
]

for a, b, c in lists:
    if 1 <= a <= 12 and 0 <= b <= 59 and 0 <= c <= 59:
        cnt += 1

print(cnt)
```

## 이 코드에서 내가 하려고 했던 것

처음 코드는 문제를 아주 정직하게 풀고 있다.

- 입력으로 받은 세 숫자를 `h, m, s`라는 이름으로 일단 꺼내 둔다.
- 가능한 배치를 6개 튜플로 직접 나열한다.
- 각 튜플에서 첫 번째 값을 시, 두 번째 값을 분, 세 번째 값을 초라고 보고 범위를 검사한다.
- 조건을 만족하면 `cnt`를 증가시킨다.

즉, "가능한 모든 해석을 다 써보고, 맞는 것만 세자"라는 브루트포스 아이디어를 그대로 코드로 옮긴 풀이였다.

이 방식은 실제로 맞는 접근이고, 공식 예제

- `00:00:00 -> 0`
- `01:00:00 -> 2`
- `01:12:23 -> 4`

에서도 기대한 결과가 나온다.

## 아쉬웠던 점 또는 반례

이번 코드는 크게 틀린 부분이 있는 풀이로 보이지는 않았다.
특별한 반례를 찾기보다는, **맞는 코드를 조금 더 정리할 수 있는 지점**이 보였다.

아쉬웠던 점은 대략 이 정도다.

- 가능한 6가지 경우를 손으로 직접 적어서, 문제는 풀리지만 코드가 조금 장황하다.
- `time`, `lists`처럼 의미가 넓은 변수명보다 `parts`, `cases`처럼 역할이 드러나는 이름이 더 읽기 쉽다.
- 범위 검사 로직을 함수로 분리하면 "유효한 시각인지 확인한다"는 의도가 더 분명해진다.

즉, 이번 문제는 로직 오류를 찾기보다는 **같은 정답 풀이를 더 읽기 좋게 다듬는 연습**에 가까웠다.

## 개선 풀이 1

기존 아이디어를 가장 자연스럽게 정리하면 `itertools.permutations`를 쓰는 방식이다.

```python
from itertools import permutations

parts = list(map(int, input().split(":")))
answer = 0

def is_valid(hour, minute, second):
    return 1 <= hour <= 12 and 0 <= minute <= 59 and 0 <= second <= 59

for hour, minute, second in permutations(parts, 3):
    if is_valid(hour, minute, second):
        answer += 1

print(answer)
```

### 설명

- 직접 6개 경우를 적는 대신 `permutations(parts, 3)`으로 모든 배치를 만든다.
- `is_valid()` 함수로 시/분/초 범위 검사 의도를 드러냈다.
- 원래 코드와 계산 방식은 같지만, 읽을 때 "무엇을 하고 있는지"가 더 잘 보인다.

원본 풀이를 거의 그대로 유지하면서, 구현만 조금 더 정돈한 버전이라고 볼 수 있다.

## 개선 풀이 2

이 문제는 사실 6가지를 전부 생성하지 않고도 더 짧게 셀 수 있다.

```python
nums = list(map(int, input().split(":")))

if not all(0 <= x <= 59 for x in nums):
    print(0)
else:
    hour_candidates = sum(1 <= x <= 12 for x in nums)
    print(hour_candidates * 2)
```

### 설명

- 세 숫자 중 하나라도 `59`를 넘으면 분이나 초가 될 수 없어서 전체 경우의 수는 `0`이다.
- 그다음에는 어떤 칸이 `시(1~12)`가 될 수 있는지만 세면 된다.
- 시가 될 수 있는 칸 하나를 고르면, 나머지 두 칸은 분과 초로 배치하는 방법이 항상 2가지 있다.
- 그래서 `시 후보 개수 * 2`가 바로 정답이 된다.

예를 들어 `01:12:23`이라면

- `01`, `12` 두 칸이 시 후보이고
- 각각에 대해 나머지 두 칸을 분/초로 바꾸는 방법이 2가지씩 있어서
- 총 `2 * 2 = 4`가지가 된다.

문제 성질을 한 번 더 이용한 풀이여서, 개인적으로는 이 버전도 꽤 재미있었다.

## 시간 복잡도

- 내가 처음 작성한 풀이: `O(1)`
- 개선 풀이 1: `O(1)`
- 개선 풀이 2: `O(1)`

이유는 세 풀이 모두 입력 숫자가 항상 3개로 고정되어 있고, 확인하는 경우의 수 역시 최대 6가지로 고정되어 있기 때문이다.

즉, 입력 크기 `n`이 커질수록 반복 횟수가 늘어나는 구조가 아니라

- 처음 풀이: 6개의 배치를 검사
- 개선 풀이 1: 순열 6개를 검사
- 개선 풀이 2: 숫자 3개만 확인

하는 식으로 항상 같은 수의 연산만 수행한다.
그래서 모두 `O(1)`이라고 볼 수 있다.

공간 복잡도도 세 숫자와 몇 개의 변수 정도만 쓰므로 모두 `O(1)`이다.

## 이번 문제에서 배운 점

- 경우의 수가 적다면 겁먹지 말고 전부 써보는 브루트포스가 가장 빠른 정답이 될 수 있다.
- 구현 문제에서는 정답 여부뿐 아니라, 범위 조건을 코드에 얼마나 또렷하게 드러내는지도 중요하다.
- 같은 문제도 "모든 경우를 직접 검사하는 방법"과 "문제 성질을 이용해 바로 세는 방법" 두 가지 관점으로 다시 볼 수 있다.
