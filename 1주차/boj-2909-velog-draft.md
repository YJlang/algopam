알고팜 1주차에서 풀었던 **BOJ 2909 - 캔디 구매** 문제를 정리해보려고 한다.  
이번 문제는 길게 구현하는 문제라기보다, **반올림을 정수 연산으로 어떻게 표현할지**가 핵심인 문제였다.

문제 링크: https://www.acmicpc.net/problem/2909

## 문제 정보

- 시간 제한: 1초
- 메모리 제한: 128MB
- 풀이 감각: 구현 / 수학 / 반올림

## 문제 설명

문제 원문은 위 링크에서 확인할 수 있고, 아래는 풀이를 위해 다시 정리한 내용이다.

- 사탕 가격 `C`가 주어진다.
- 상근이는 `1, 10, 100, 1000, ...` 형태의 특정 액면가 지폐만 가지고 있다.
- 그 지폐의 액면가는 `10^K`원이다.
- 거스름돈은 받지 않기 때문에, 사장은 사탕 가격을 **상근이가 낼 수 있는 가장 가까운 금액**으로 반올림해준다.
- 결국 `C`를 `10^K`의 배수 중 가장 가까운 값으로 바꾸면 된다.

예를 들어 지폐 단위가 `100`원일 때

- `150`원은 `200`원으로
- `149`원은 `100`원으로

바뀐다.

## 입력 형식

- 첫째 줄에 사탕 가격 `C`와 0의 개수 `K`가 주어진다.
- 상근이가 가지고 있는 지폐의 액면가는 `10^K`원이다.
- `0 <= C <= 1,000,000,000`, `0 <= K <= 9`

## 출력 형식

- 상근이가 실제로 내야 하는 가격을 출력한다.
- 즉, `C`를 `10^K` 단위로 반올림한 결과를 출력하면 된다.

## 예제 입력 1

```text
184 1
```

## 예제 출력 1

```text
180
```

## 예제 입력 2

```text
123450995 1
```

## 예제 출력 2

```text
123451000
```

## 예제 입력 3

```text
182 2
```

## 예제 출력 3

```text
200
```

## 접근 아이디어

이 문제는 결국 한 줄로 요약하면 아래와 같다.

1. 지폐 단위 `10^K`를 구한다.
2. 가격 `C`를 그 단위에 맞춰 반올림한다.
3. 반올림된 값을 출력한다.

`10` 단위 반올림, `100` 단위 반올림, `1000` 단위 반올림을 일반화한 문제라고 생각하면 이해가 쉽다.

## 내가 처음 작성한 코드

제출 당시 코드는 아래 흐름이었다.  
블로그에는 같은 로직을 유지하되, 주석만 조금 정리해서 가져왔다.

```python
import sys

input = sys.stdin.readline

c, k = map(int, input().split())
sanggeun_money = 10 ** k

print(((c + sanggeun_money // 2) // sanggeun_money) * sanggeun_money)
```

## 이 코드에서 내가 하려고 했던 것

핵심 아이디어는 아주 단순하다.

- `10^K`를 기준 단위로 잡는다.
- 반올림을 위해 기준 단위의 절반을 먼저 더한다.
- 그다음 기준 단위로 나눈 뒤 다시 곱하면, 가장 가까운 배수로 맞춰진다.

예를 들어 `C = 184`, `K = 1`이면

- 기준 단위는 `10`
- 절반은 `5`
- `184 + 5 = 189`
- `189 // 10 = 18`
- `18 * 10 = 180`

이런 식으로 자연스럽게 `10` 단위 반올림이 된다.

## 아쉬웠던 점

이번 코드는 결과 자체는 맞는 풀이였다.  
다만 블로그 글로 설명하려고 다시 보니, 아래 부분은 조금 아쉬웠다.

- 식이 짧아서 처음 보면 왜 맞는지 바로 읽히지 않는다.
- `sanggeun_money`는 실제 가진 돈의 양이라기보다 **지폐 단위**에 가까운 이름이다.
- 이런 문제에서 파이썬의 `round()`를 떠올리기 쉬운데, 그 방식은 오히려 위험할 수 있다.

특히 Python의 `round()`는 우리가 보통 생각하는 무조건적인 사사오입과 다르게, **banker's rounding** 방식이 섞여 있어서 조심해야 한다.

예를 들어 `10` 단위 반올림을 `round()`로 처리하면

```text
25 1
```

같은 경우는 문제 기준으로는 `30`이 되어야 하지만, Python에서 `round(25, -1)`은 `20`이 된다.  
그래서 이 문제는 현재 코드처럼 **정수 연산으로 직접 처리하는 방식**이 훨씬 안전하다.

## 개선 풀이 1: 같은 아이디어를 조금 더 읽기 쉽게

원래 방식 자체는 좋기 때문에, 변수 이름과 중간 의미만 조금 더 분명하게 하면 훨씬 읽기 편해진다.

```python
import sys

input = sys.stdin.readline

price, k = map(int, input().split())
unit = 10 ** k

rounded_price = ((price + unit // 2) // unit) * unit
print(rounded_price)
```

### 설명

- `unit`은 반올림 기준이 되는 지폐 단위다.
- `unit // 2`를 더해 반올림 기준을 맞춘다.
- 이후 `// unit`, `* unit`을 통해 가장 가까운 배수로 만든다.

### 이 풀이가 좋은 이유

- 원래 아이디어를 그대로 살린다.
- 수식이 짧고 빠르다.
- 시간 복잡도는 `O(1)`이다.

## 개선 풀이 2: 반올림 기준을 더 드러내고 싶다면

수식을 한 번에 쓰는 대신, 몫과 나머지로 나누어 생각하면 반올림 기준이 더 눈에 잘 들어온다.

```python
import sys

input = sys.stdin.readline

price, k = map(int, input().split())
unit = 10 ** k

quotient, remainder = divmod(price, unit)

if remainder * 2 >= unit:
    quotient += 1

print(quotient * unit)
```

### 이 방식의 포인트

- `quotient`는 현재 몇 개의 `unit`이 들어가는지
- `remainder`는 얼마나 남는지를 뜻한다.
- 남는 값이 절반 이상이면 올리고, 아니면 그대로 둔다.

개인적으로는 제출용으로는 첫 번째 개선 풀이가 더 짧고 좋고,  
설명용으로는 두 번째 풀이가 반올림 원리를 보여주기 더 좋다고 느꼈다.

## 이번 문제에서 배운 점

- 반올림 문제는 실수 연산보다 **정수 연산으로 처리하는 편이 안전한 경우가 많다.**
- `10^K` 같은 기준 단위를 먼저 명확하게 잡으면 문제가 훨씬 단순해진다.
- 짧은 코드가 항상 나쁜 것은 아니지만, **왜 맞는지 설명할 수 있을 정도로 구조를 이해하고 있는가**도 중요하다.

이번 문제는 분량은 짧았지만,  
오히려 그래서 한 줄짜리 수식이 어떤 의미를 가지는지 곱씹어보기 좋은 문제였다.
