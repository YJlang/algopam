알고팜 1주차에서 풀었던 **BOJ 22993 - 서든어택 3**를 정리해보려고 한다.  
이 문제는 전투를 직접 시뮬레이션하는 것처럼 보이지만, 실제로는 **정렬 + 그리디**로 생각하면 훨씬 깔끔하게 풀리는 문제였다.

문제 링크: https://www.acmicpc.net/problem/22993

## 문제 정보

- 시간 제한: 2초
- 메모리 제한: 1024MB
- 풀이 감각: 그리디 / 정렬 / 시뮬레이션

## 문제 설명

문제 원문은 위 링크에서 확인할 수 있고, 아래는 풀이를 위해 다시 정리한 내용이다.

- 플레이어가 총 `N`명 있고, 준원이는 1번 플레이어다.
- 준원이의 초기 공격력은 `A1`, 나머지 플레이어의 공격력은 `A2 ... AN`이다.
- 공격력이 더 높은 쪽이 이기고, 이긴 플레이어의 공격력은 두 사람의 공격력 합으로 증가한다.
- 공격력이 같으면 아무 일도 일어나지 않는다.
- 전투 순서를 잘 정해서 **준원이만 최후의 생존자**가 될 수 있으면 `Yes`, 아니면 `No`를 출력하면 된다.

이 문제의 포인트는 결국 아래 질문으로 바뀐다.

- 준원이가 다른 플레이어들을 어떤 순서로 만나야 계속 살아남을 수 있을까?

## 입력 형식

- 첫째 줄에 플레이어 수 `N`이 주어진다.
- 둘째 줄에 각 플레이어의 공격력 `A1, A2, ..., AN`이 공백으로 주어진다.
- `1 <= N <= 100000`
- `1 <= Ai <= 1000000000`

## 출력 형식

- 좋은 전투 순서가 존재해서 준원이만 살아남을 수 있으면 `Yes`
- 어떤 순서로 하더라도 불가능하면 `No`

## 예제 입력 1

```text
5
2 1 2 3 4
```

## 예제 출력 1

```text
Yes
```

## 예제 입력 2

```text
5
2 1 1 1 10
```

## 예제 출력 2

```text
No
```

## 접근 아이디어

처음 이 문제를 보자마자 든 생각은 이거였다.

1. 지금 이길 수 있는 상대를 먼저 잡는다.
2. 그러면 공격력이 커진다.
3. 공격력이 커지면 원래 못 이기던 상대도 이길 수 있게 될 수 있다.

즉, **이길 수 있는 상대 중 가장 약한 상대부터 잡는 것**이 가장 유리하다.

왜냐하면

- 더 약한 상대는 지금 바로 이길 수 있고
- 그 상대를 잡아 공격력을 조금이라도 키우면
- 다음 단계에서 이길 수 있는 상대의 범위가 넓어지기 때문이다.

그래서 남은 적들을 공격력 기준으로 오름차순 정렬한 뒤, 앞에서부터 하나씩 먹을 수 있는지만 보면 된다.

## 내가 처음 작성한 코드

```python
n = int(input())
arr = list(map(int, input().split()))

def sudden():
    jun = arr.pop(0)
    arr.sort()

    for i in arr:
        if jun > i:
            jun += i
        else:
            return "No"
    return "Yes"

print(sudden())
```

## 이 코드에서 내가 하려고 했던 것

현재 코드는 아래 흐름으로 동작한다.

1. `arr[0]`을 준원이의 공격력으로 꺼낸다.
2. 나머지 플레이어 공격력을 오름차순 정렬한다.
3. 정렬된 순서대로 보면서, 지금 준원이의 공격력으로 이길 수 있으면 흡수한다.
4. 하나라도 못 이기면 즉시 `No`를 출력한다.
5. 끝까지 모두 이기면 `Yes`다.

예를 들어 공격력이 아래처럼 있다고 해보자.

```text
2 / 1 2 3 4
```

그러면

- 2로 1을 이겨서 3
- 3으로 2를 이겨서 5
- 5로 3을 이겨서 8
- 8로 4를 이겨서 12

가 되므로 최종적으로 전부 이길 수 있다.

## 왜 이 그리디가 맞는가

이 문제에서 가장 중요한 부분은 “정렬해서 약한 적부터 처리하는 게 왜 맞는가”이다.

남아 있는 적 중에서 **가장 약한 적도 못 이기면**, 그보다 강한 적은 당연히 더 못 이긴다.  
즉, 그 시점에서 더 이상 진행할 방법이 없으므로 정답은 `No`가 된다.

반대로 가장 약한 적을 이길 수 있다면, 그 적을 먼저 잡는 것이 손해일 이유가 없다.

- 승리하면 공격력이 증가하고
- 다음 전투에서 선택 가능한 적이 줄어들지 않고 오히려 늘어나기 때문이다.

그래서 이 문제는

- 남은 적을 오름차순으로 보고
- 현재 공격력보다 작은 적을 차례대로 흡수하는 방식

으로 충분하다.

## 아쉬웠던 점

이번 코드는 정답 풀이로는 충분히 좋았다.  
다만 다시 보니 조금 더 다듬을 수 있는 부분은 있었다.

- `arr.pop(0)`은 리스트 맨 앞 원소를 꺼내기 때문에 내부적으로 원소를 한 칸씩 당긴다.
- 물론 이 문제에서는 정렬 `O(N log N)`이 더 크기 때문에 치명적이지는 않지만, 굳이 리스트를 변형하지 않아도 된다.
- 함수 이름 `sudden`, 변수 이름 `jun`, `i`도 제출용으로는 괜찮지만 블로그 설명용으로는 의미가 조금 더 드러나면 좋다.

또 하나 중요한 포인트는 비교가 `jun > i`라는 점이다.

- 문제에서 공격력이 같으면 아무 일도 일어나지 않는다.
- 따라서 `>=`가 아니라 반드시 `>`여야 한다.

이 부분을 놓치면 오답이 날 수 있다.

## 개선 풀이 1: 같은 아이디어를 더 읽기 쉽게

```python
import sys

input = sys.stdin.readline

n = int(input())
attacks = list(map(int, input().split()))

junwon = attacks[0]
enemies = sorted(attacks[1:])

for enemy in enemies:
    if junwon <= enemy:
        print("No")
        break
    junwon += enemy
else:
    print("Yes")
```

### 설명

- `junwon`은 현재 준원이의 공격력
- `enemies`는 나머지 플레이어를 정렬한 리스트다.
- 가장 약한 적부터 보면서 이길 수 있으면 흡수하고, 못 이기면 바로 종료한다.

원래 코드와 아이디어는 같지만,  
리스트를 직접 수정하지 않고 의미가 더 잘 드러나도록 바꾼 버전이다.

## 개선 풀이 2: 함수형으로 분리해서 쓰고 싶다면

```python
import sys

input = sys.stdin.readline

def can_survive_all(attacks):
    power = attacks[0]

    for enemy in sorted(attacks[1:]):
        if power <= enemy:
            return False
        power += enemy

    return True

n = int(input())
attacks = list(map(int, input().split()))

print("Yes" if can_survive_all(attacks) else "No")
```

### 이 방식의 포인트

- 핵심 로직을 `can_survive_all`로 분리해서 의도를 더 분명하게 드러낸다.
- 테스트하거나 재사용하기도 더 편하다.
- 본질적인 그리디 아이디어는 동일하다.

## 시간 복잡도

- 내가 처음 작성한 풀이: `O(N log N)`
- 개선 풀이 1: `O(N log N)`
- 개선 풀이 2: `O(N log N)`

왜 이렇게 되는지도 같이 보면 명확하다.

- 가장 큰 비용은 적들의 공격력을 정렬하는 단계다.
- 정렬에 `O(N log N)`이 걸린다.
- 이후 한 번 순회하면서 흡수 여부를 확인하는 부분은 `O(N)`이다.
- 따라서 전체 시간 복잡도는 정렬이 지배하므로 `O(N log N)`이 된다.

추가로 원래 코드의 `pop(0)`은 리스트 맨 앞 삭제라서 `O(N)`이지만,  
이 역시 정렬 비용보다 작아서 전체 복잡도는 그대로 `O(N log N)`이다.

## 이번 문제에서 배운 점

- 전투 순서를 직접 시뮬레이션하기보다, “어떤 순서가 가장 유리한가”를 먼저 생각하는 게 중요하다.
- 현재 이길 수 있는 적 중 가장 약한 적부터 처리하는 그리디가 자연스럽게 성립한다.
- 그리디 문제는 단순히 “정렬해서 한다”가 아니라, **왜 그 선택이 손해가 아닌지**를 설명할 수 있어야 더 탄탄하게 이해한 것이다.

이번 문제는 구현 자체는 짧았지만,  
그리디 정당성을 한 번 더 점검하기 좋은 문제였다.

## 마무리

처음 작성한 코드도 핵심 아이디어를 잘 잡고 있어서 충분히 좋은 풀이였다.  
다만 다시 정리해보면서,

- 왜 정렬이 필요한지
- 왜 가장 약한 적부터 처리해야 하는지
- 비교를 왜 `>`로 해야 하는지

까지 같이 적어두니 훨씬 더 또렷하게 남는 느낌이었다.
